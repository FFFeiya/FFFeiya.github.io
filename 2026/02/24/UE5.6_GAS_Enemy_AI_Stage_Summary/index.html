用 UE5.6 + GAS 搭一个可迭代的敌人近战 AI：从状态机图到可调试行为树

关键词：Unreal Engine 5.6、Gameplay Ability System（GAS）、Behavior Tree、Blackboard、Motion Warping、近战 HitScan、GameplayEffect、调试与工程化

1. 背景：我在做什么

项目里同事已经基于 GAS 搭好了「基础角色」「受击木桩」「近战武器与命中窗口（HitScan Window）」等战斗底座。我需要在初学 GAS的前提下，做一个敌人 AI，能按设计图执行：

  • 距离近：轻击 → 轻击连招（先做一招/后续可扩展）→ 后退拉开距离
  • 距离远：追击/位移 → 靠近后轻击
  • 按节奏循环，且能持续调试、扩展到更多招式/更多状态

我把这件事拆成两条主线：

  1. AI 决策：把图里的流程翻译为 Behavior Tree（BT）+ Blackboard（BB）
  2. 战斗触发：让 BT 触发 GAS 的能力（GA），并接上现有武器 HitScan/伤害体系

2. 目标与验收标准

2.1 验收（可见行为）

  • 敌人能锁定目标（玩家/木桩）
  • 能随距离切换：追击 / 近战攻击 / 后撤
  • 近战攻击走 GAS：播放 Montage、触发 HitScan、对目标扣血
  • 不会出现「攻击一次后 AI 永久卡死」或「一直 Wait 不动」等锁死现象

2.2 工程化验收（可维护）

  • 关键状态（Target、Distance、IsAttacking、RetreatLocation）都可在 Blackboard 里观察
  • Task/Service 有足够的打印与失败分支，避免“静默失败”
  • 能快速替换攻击能力（GA_Enemy_LightCombo → 其他 GA），不用重写 AI

3. 总体结构：BT/BB + GAS + 武器 HitScan

整体结构可以理解为三层：

3.1 行为树层（AI 决策）

  • BB_Enemy：保存 AI 当前“世界观”
  • BT_Enemy：按距离选择 Chase/CloseAttack 分支
  • BTService：每 0.2 秒更新距离、目标位置、后撤点
  • BTTask：触发攻击开始/结束（以及可选的连招输入）

3.2 能力层（GAS）

  • GA_Enemy_LightCombo：敌人轻击能力（继承 GA_MeleeAttack_Base）
  • 能力播放 Montage，并通过 GameplayEvent 打开/关闭 HitScan 窗口

3.3 武器层（命中检测 + 伤害应用）

  • BP_Weapon_Base / BP_Weapon_EnemyKnife：在 HitScan 窗口内做 Trace，找到命中目标，把 GameplayEffect 应用到目标 ASC（AbilitySystemComponent）

4. 资源清单（我创建/改动了哪些文件）

命名以我的工程里实际资源为准（面试官问的时候我会按这一套展开）。

4.1 AI 资源

  • BB_Enemy（Blackboard Key）
    • TargetActor (Object) / TargetLocation (Vector)
    • DistToTarget (Float)
    • HasTarget (Bool)
    • IsAttacking (Bool) —— 用“是否设置”模拟状态门禁
    • RetreatLocation (Vector)
  • BT_Enemy（Behavior Tree）
  • BTService_BlueprintBase（Service：每 0.2s 更新 BB）
  • BP_EnemyAIController（OnPossess 运行 BT，并初始化 BB）

4.2 BT Tasks

  • BTT_LightStart：尝试激活 GA（Try Activate Ability by Class）
  • BTT_AttackEnd：清理攻击门禁（Clear Value(IsAttacking)）
  • （可选）BTT_ComboContinue：发送连招输入事件（后续扩展用）

4.3 GAS / 动画 / 武器

  • GA_Enemy_LightCombo（data-only，继承 GA_MeleeAttack_Base）
  • AM_Enemy_LightCombo（Montage，含 Hit Scan Window NotifyState）
  • BP_Weapon_EnemyKnife（武器配置）
  • BP_Weapon_Base（HitScan 与 ApplyEffect 逻辑修复）

5. 把“流程图”翻译成 BT：我怎么拆

设计图的核心是“距离判断 + 一套近战循环”。
我把它映射成一个典型的 BT 结构：

  • Root → Selector
    • Combat 分支（Decorator：TargetActor is Set）
      • Sequence（挂 Service）
        • Selector（按距离）
          • CloseAttack（Dist <= 300 且 IsAttacking Not Set）
            • LightStart（触发攻击）
            • Wait（给蒙太奇/窗口时间）
            • AttackEnd（清门禁，避免锁死）
            • MoveTo RetreatLocation（后撤）
          • Chase（Dist > 300 且 IsAttacking Not Set）
            • MoveTo TargetActor（追击）
            • Wait（防抖）
            • LightStart
            • AttackEnd
    • Fallback Wait（没目标就空转）

5.1 关于 IsAttacking：为什么不用 True/False，而用 Set/Not Set

我在 UE5.6 的 BT Decorator 菜单里，只能选 Is Set / Is Not Set（没有“Is True/Is False”比较版本）。
所以我把 IsAttacking 作为“门闩”来用：

  • 攻击开始：SetBB(IsAttacking)=true（= 已设置）
  • 攻击结束：Clear Value(IsAttacking)（= 未设置）

这样 Decorator IsAttacking is Not Set 就等价于 “IsAttacking == false”。

这个决策的价值：

  • 兼容现有工具限制
  • 逻辑清晰
  • 解决“攻击一次后永远卡死”的典型问题（后面会细讲）

6. 关键实现细节与踩坑记录

6.1 TryActivate 一直 False：原来是“能力根本没授予”

我在 Task 里用 Try Activate Ability by Class(GA_Enemy_LightCombo)，但一直返回 False。
用 Find All Abilities with Tags + Length 打印发现 AbilityCount=0，说明敌人的 ASC 里压根没有这条能力 Spec。

根因：我只是把 GA 填进 Abilities to Grant 属性，但项目里并没有自动遍历授予（或授予链路没跑）。
修复：在敌人 BeginPlay / 初始化链路里显式 Give Ability(GA_Enemy_LightCombo)，验证后 AbilityCount=1，TryActivate 变 True。

我不是盲猜“权限/Tag/冷却”，而是先验证“Spec 是否存在”，把问题从“CanActivate”缩小成“授予链路”。

6.2 TargetActor 变 invalid：Service 的“数据源”要统一

黑板里经常出现 TargetActor (invalid)，导致 TargetLocation/RetreatLocation invalid，距离恒为 0，BT 回退到 Root Wait。

修复思路：Service 每 tick 不再“信任黑板旧引用”，而是用当前帧的 GetPlayerCharacter 作为唯一数据源：

  • 先写回 BB(TargetActor/HasTarget)
  • 再用同一个对象算 TargetLocation/DistToTarget
    这样避免引用残留/时序导致的 invalid。

这是典型“数据一致性”问题：要么全从黑板读，要么全从实时来源算。混用会产生不可预期状态。

6.3 攻击一次后无限 Wait：MoveTo 卡住导致 AttackEnd 永远不执行

出现过典型锁死：第一次 LightStart 成功把 IsAttacking 设为 true，但 AttackEnd 没执行到，导致 Close/Chase 的 Decorator 永远 fail。

根因：MoveTo RetreatLocation 可能因为目标点动态变化、或在 NavMesh 外导致一直不结束。
修复：把顺序改为：

  • 先 AttackEnd（Clear IsAttacking）再 MoveTo Retreat
    即使 MoveTo 卡住，也不会把 AI 的“攻击门禁”永久锁死。

我把“关键状态释放”与“可能失败/耗时的移动”解耦，提升系统鲁棒性。

6.4 敌人打玩家扣自己血：武器 HitScan 里忽略对象写死成玩家

伤害走 GA_MeleeAttack_Base → HandleHitScanStart → 传 EffectSpec 给 BP_Weapon_Base。
真正 Apply 发生在武器里。

我发现武器 Trace 的 Actors to Ignore 里写死了 GetPlayerCharacter(0)（为玩家武器设计），当武器给 AI 用时就会：

  • 忽略玩家 → 打不到玩家
  • 不忽略 Owner → 更可能命中自己
    修复：改成忽略 GetOwner()（武器持有者），并额外加“HitActor==Owner 跳过”保险。

这是典型“复用系统从 Player 扩展到 AI”时的隐性耦合问题：把数据从“固定玩家”抽象成“Owner/Instigator”。

8. 我现在的阶段成果

8.1 我做出了什么

  • 用 BT/BB/Service/Task 把“距离驱动的近战 AI”跑通
  • 敌人能通过 GAS 激活近战能力，播放 Montage，并触发 HitScan 窗口
  • 修复了“授予链路缺失”“目标 invalid”“攻击状态锁死”“武器忽略对象写死玩家”等问题
  • 整套系统可扩展到更多招式（只要换 GA 与 Montage/Notify）

8.2 我用什么方法解决问题

  • 先 观测：Blackboard/Print/AbilityCount/HitActorName
  • 再 定位：把问题拆成“AI 决策”“能力授予/激活”“命中检测/Apply”三层
  • 最后 工程化修复：加失败分支 Finish、把状态释放放在 MoveTo 前、统一数据源

8.3 QA

  • Q：为什么 TryActivate 一直失败？
    A：我先验证 Spec 是否存在。用 Find All Abilities with Tags 打印 AbilityCount=0，发现不是 CanActivate 条件，而是能力根本没授予。补 GiveAbility 后成功。
  • Q：为什么会出现 AI 打自己？
    A：伤害 Apply 在武器层；武器 Trace 忽略对象写死为玩家，AI 拿武器时命中自己。修复为 ignore Owner，并过滤 HitActor==Owner。
  • Q：为什么会出现“攻击一次后卡死”？
    A：IsAttacking 门禁没释放；MoveTo/动态目标可能卡住导致 AttackEnd 不执行。我把状态释放提前到 MoveTo 前，避免锁死。

9. 下一步计划（可继续迭代）

  • 把轻击连招扩展到 2 段（ContinueComboInput 或 Montage section + event）
  • RetreatLocation 使用 Project Point to Navigation 投射到 NavMesh，后撤更稳定
  • 接入 AI Perception（视线/听觉）替代“永远锁玩家”为目标
  • 把关键参数（近战距离、后撤距离、攻击冷却）做成 DataAsset/可调表

10. 小结

这次工作表面上是在“拼 BT 节点”，本质上是在做三件事：

  1. 把设计图翻译成可调试的状态机（BT+BB）
  2. 把 AI 的“决策”与 GAS 的“执行”正确对接
  3. 用工程化手段消灭“隐性耦合”（玩家写死、引用 invalid、状态锁死）

最终得到的是一套可复用、可扩展、能讲清楚的敌人近战 AI 方案。

本文为阶段性总结，后续会继续补充：二段连招、更多敌人 archetype、AI 感知与战斗节奏。这个是与同学合作制作的动作游戏的阶段性总结。
